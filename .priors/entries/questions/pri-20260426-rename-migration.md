## What's known

- `.priors/project.json` holds a UUID that is the canonical identity. Tests verify identity survives directory rename within a single machine.
- MCP client configs generated by `priors init-config` pin the project root path. After a rename, the user must regenerate or hand-edit these.

## What's not yet decided

- Should `priors init-config` write a small symlink or pointer file that resolves the project by UUID, so client configs survive a `mv`?
- Should `priors health` warn when the working directory's path doesn't match the path stored in client configs?
- Is there a `priors relocate` verb that updates known external references atomically?

## Resolution path

This blocks no v1 work. Resolve before shipping `priors init-config` polish (Phase 7). If unresolved at v1.0, document the manual regeneration step in `docs/integrations.md`.
